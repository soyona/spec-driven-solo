# 📘 Spec-Driven Solo 新手入门指南(V2.1.0-Kernel)

> **最高导言**：本指南是用冰冷的工程法律，去约束并释放多元化平台下的 AI 生产力。请坐在主驾驶位上，冷酷地执行每一次 Review，绝对不要纵容 AI 的任何一次任性。

## 🟢 第一章：核心心法与思维重构 (The Mindset)

本章确立你与 AI 智能体（Agent）协作时的底层法理与思维边界。认知一旦跑偏，任何工具链都会退化为“修不完的 Bug 死循环”。

### 1.1 两个角色的绝对切割 (Role Separation)

在 Spec-Driven 范式下，你不再是写代码的“码农”，你的身份必须发生绝对重构：

```text
  【你：架构师 / 严苛的审阅者 (CTO)】
                │
                ▼ (下发：高纯度技术契约)
  【AI：高并发、无情的肉体搬砖手 (Agent)】

```

* **人类的职责**：出脑子、定边界、锁死强类型数据契约、审查 Visual Diff（代码改动差异）、运行测试、果断按下熔断停止键。
* **AI 的职责**：严格基于内存银行（Memory Bank）规范，进行高并发的代码编写、依赖安装、自动化纠错与逻辑落地。
* **铁律**：**永远不要让 AI 帮你做高层决策，也永远不要替 AI 手敲具体实现代码。**

### 1.2 为什么传统流程必死：AI 时代的“代码逆向涌现”

传统 Solo 开发是“边想边写”，遇到 Bug 现场查。这种模式在面对 AI Agent 时会引发灾难性的“幻觉风暴”：

1. 你的一句模糊指令（如：“加个用户列表”）会迫使 AI 开始猜测你的意图。
2. AI 为了迎合你，会自作主张引入未经测试的第三方库，导致系统熵增。
3. 随着多轮对话消耗 Token，AI 会开始遗忘最初的架构，导致代码文件前后矛盾，陷入“报错 $\rightarrow$ 瞎猜修复 $\rightarrow$ 引入新报错”的死循环。

**新范式核心**：通过 `memory-bank/` 锁死上下文。代码不是写出来的，而是基于完美的契约“逆向涌现”出来的。

### 1.3 $20/月的资产边界与额度榨取

本指南的核心商业目标是：**拒绝购买任何昂贵的 API Token，彻底榨取每月 $20 的 ChatGPT Plus 网页端与 Codex 桌面端的免费性能极限。**

* **高级军师（ChatGPT Plus 网页端）**：专门用来干“脏活、累活、大局观活”。利用 **Thinking 模型**的深度推理能力，帮你把模糊的产品资产（PRD）翻译成严密的 `dataModels.md` 和架构图纸。它负责思考。
* **前线特种兵（Codex / Cline 桌面端）**：专门用来干“体力活”。将网页端生成的 Spec 丢入本地，利用桌面端的自动读写能力挂机搬砖。它负责执行。
* **协作边界**：不要让桌面端 Agent 去想高层架构，它的智商不够；不要让网页端去手写几千行代码，复制粘贴会断。

## 🟡 第二章：研发阵地初始化与工程化契约标准 (Infrastructure & Contracts)

本章确立项目启动时的物理防线，规范你如何通过静态约束与动态记忆锁死 AI 的幻觉。

### 2.1 极简工具链的绝对联动线 (The Toolchain Chain)

不要配置复杂的自动化流水线，只锁死以下三位一体的黄金链路：

1. **ChatGPT Plus (Web 浏览器端)**：作为“云端总设计师”。使用高级 Thinking 模型，专门负责解析原始 PRD 并输出强类型契约。
2. **VS Code (IDE 客户端)**：作为“人机协同物理沙盒”。安装 **Codex/Cline** 扩展，并打开终端面板，作为唯一的执行现场。
3. **Codex/Cline (Agent 运行端)**：作为“全自动搬砖手”。配置为 `Plan/Act` 双模式运行，拥有本地文件读写与终端命令执行权限。

---

### 2.2 独行侠的“工程化契约”语法标准 (Spec Syntax Standards)

行业虽然没有形成统一结论，但你在编写 `memory-bank/` 时必须遵守以下硬性工程语法标准，彻底终结自然语言的二义性：

#### 规范一：数据格式“强类型优先 (Types First)”

严禁在 Spec 中使用“用户包含名字和状态”等文学性描述。在 `memory-bank/dataModels.md` 中，**必须直接使用标准 TypeScript Type 或 JSON Schema 锁死数据契约**：

```markdown
### [契约 01] - 核心用户模型 (UserModel)
- **定义文件**: `src/types/user.ts`
- **契约源码**:
```typescript
export interface User {
  id: string;               // UUID 规范: crypto.randomUUID()
  email: string;            // 必须包含 @ 符号的前端校验
  role: 'admin' | 'user';   // 严格的字面量联合类型
  createdAt: string;        // ISO 8601 字符串格式 (YYYY-MM-DDTHH:mm:ssZ)
}

```

#### 规范二：组件描述“入参与副作用锁死 (Props & Side-Effects)”

在 `memory-bank/systemPatterns.md` 中描述 UI 组件树时，必须以表格或列表形式强制声明组件的 **输入 (Props)**、**输出 (Events)** 和 **副作用 (Side-Effects)**：

```markdown
### [组件 02] - UserTable.tsx
- **路径**: `src/components/users/UserTable.tsx`
- **硬性约束**:
  - `Props`: `users: User[]` (必须复用 [契约 01])
  - `Events`: `onDelete: (id: string) => void`
  - `Side-Effects`: 严禁在此组件内直接发起 fetch 请求，所有数据必须由父级容器组件下发。

```

#### 规范三：声明“严禁引入的依赖黑名单 (Negative Constraints)”

在 `memory-bank/techContext.md` 中，必须明确指出 AI 绝对不能踩的红线。AI 的惯性思维是“遇到问题就装新包”，必须用黑名单进行防御：

```markdown
### [黑名单与硬约束]
1. 状态管理：严禁安装 `redux`, `zustand`, `mobx`。所有状态必须收拢在 React Context 中。
2. 网络请求：严禁安装 `axios`。必须统一使用原生的原生 `fetch` API，并封装在 `src/lib/api.ts` 中。
3. 样式方案：严禁使用 `styled-components`。所有样式必须通过 Tailwind CSS 原子类实现。

```

---

### 2.3 动态“内存银行 (Memory Bank)”的高频同步律

静态的 `spec.md` 无法应对开发中产生的临时技术债。必须通过动态记忆文件，让 AI 自我维护项目的最新状态。

1. **`activeContext.md`（防止长对话失忆补丁）**：
每当你给 Codex/Cline 下达新任务前，它必须向该文件写入当前的即时状态。
* *实例*：“当前分支正在处理用户删除逻辑，因 localStorage 无法在 SSR 阶段读取，临时采取在 `useEffect` 中挂载的权宜之计。”


2. **`progress.md`（切香肠计数器）**：
严禁一次性推进多个功能。此文件必须严格执行单线程勾选。
* `[x]` 代表已通过本地编译和人肉质检。
* `[-]` 代表当前正在 Act 模式下攻坚的文件。
* `[ ]` 代表尚未解锁的冰封特性。

## 🔴 第四章：新手必遇深坑与熔断机制 (Error Handling & Breakout)

本章是你的**生命线**。当 AI 陷入死循环、开始胡言乱语、或在同一个 Bug 上反复折腾时，本章提供的硬性指标将把你从 Token 燃烧和精神崩溃的边缘拉回来。

### 4.1 三次报错原则（硬熔断机制）

在 Solo 开发中，最容易让人上头的是“看 AI 现场修 Bug”。AI 报错后会习惯性地说 *“对不起，这是我的疏忽，我可以用以下方式修复……”*，然后写出更错的代码。

#### ⚖️ 熔断执行标准：

一旦在终端执行编译（如 `npm run build` 或 `tsc`）时，**同一个错误或同一类红字报错连续出现 3 次**，人类必须**强行介入**，严禁发送第 4 次“继续修复”的指令。

#### 🛠️ 暴力自救三步法：

1. **按下 Stop**：立刻点击 Codex/Cline 界面上的 **Stop/Cancel** 按钮，强行终止 AI 的 Act 行为。
2. **Git 暴力回滚**：不要试图在弄脏的代码堆里打补丁。直接在终端执行：
```bash
git reset --hard HEAD

```


将源码轨（`src/`）瞬间恢复到上一个干净的、可编译的 Commit 节点。
3. **重构因果链**：AI 连续 3 次修不好一个 Bug， 100% 不是因为代码写错了，而是因为 **`memory-bank/` 里的某条逻辑冲突了**，或者 `techContext.md` 里漏掉了某个关键的环境依赖。修改记忆文件，然后重新开启对话。

---

### 4.2 “丧失记忆/死循环”急救包 (Context Flashing)

当对话轮次过多，本地 Agent 触发了上下文溢出（Token Windows Full）时，它会表现出以下征兆：

* **胡言乱语**：开始修改与当前任务完全不相关的历史文件。
* **精神分裂**：上一轮对话刚商量好的技术栈，这一轮突然开始自顾自地推翻。

#### 🩹 急救清除术（Flash Context）：

1. **同步现状**：在当前的对话框输入最后一条命令：
> “请立刻停止任何代码编写。根据你目前的源码改动，将最新的项目进展、尚未解决的阻碍、以及技术债，完整写入 `memory-bank/activeContext.md`。写完后告诉我。”


2. **暴力斩断对话**：点击“新对话 (New Session / Clear Chat)”按钮，**彻底杀掉当前的上下文进程，清空内存。**
3. **通过圣经唤醒**：在新对话的第一句话，发送通读指令：
> “这是一个全新的开始。请完整阅读 `memory-bank/` 下的所有文件以重建你的大脑，并查看 `activeContext.md` 确认我们刚才停留在哪一步。理解后，请向我汇报接下来要修改的第一个文件。”

### 4.3 复杂系统下的“分布式 Spec”拆分艺术

当你的项目随着迭代越来越大，`memory-bank/` 下的 Markdown 文件行数可能会激增。如果把一个庞大系统的全局规范一次性喂给 AI，它会因为上下文太长而变得迟钝。

#### 🏗️ V1.3.0 扩容规范：

当主功能开发完毕，准备开发某个庞大的独立子系统（例如：从零接入一个极为复杂的第三方 Webhook 账单系统）时，采取“分布式外挂大脑”策略：

1. 在 `memory-bank/` 下新建一个**专属子文件夹**：`memory-bank/billing-subsystem/`。
2. 在这个文件夹内，由 ChatGPT 网页端单独压榨出一套子规范：`billing-spec.md`、`billing-models.md`。
3. 在主规范 `systemPatterns.md` 中，只写一行路由级别的抽象：
> `[模块 09] - 账单中心：其内部核心契约与流转逻辑，完全解耦并锁死在 memory-bank/billing-subsystem/ 目录中。主系统仅通过 src/types/billing.ts 进行类型通信。`



通过这种“分布式契约”的架构，你可以无上限地扩展你的产品复杂线，而每一条开发战线上的 AI 智能体都能保持极高的专注力和绝对的极速。

---

#### 🏗️ V2.0.0 矩阵式扩容规范：

当主功能开发完毕，准备开发某个极其庞大或异构的独立子系统时（例如：在你的 Mac 桌面端应用中加入一个极为复杂的 Rust 原生图像处理加速引擎），采取“矩阵式外挂大脑”策略：

1. 在 `memory-bank/` 下新建一个**专属子文件夹**：`memory-bank/rust-core/`。
2. 在该文件夹内，由 ChatGPT 网页端针对新的技术特性（Rust 轮廓）单独压榨出一套子规范：`rust-spec.md`、`cargo-models.md`。
3. 在主规范 `techContext.md` 与 `systemPatterns.md` 中，只写一行安全路由级别的抽象：
> `[模块 09] - 图像处理中心：其内部底层安全契约与 Rust Panic 拦截逻辑，完全解耦并锁死在 memory-bank/rust-core/ 目录中。主系统仅通过 Tauri IPC Bridge 进行通信。`

通过这种“矩阵式契约”的架构，你可以无上限地扩展你的产品形态，而每一条开发战线上的 AI 智能体都能保持极高的专注力和绝对的极速。
